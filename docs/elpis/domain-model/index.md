# 领域模型架构设计 - 80/20配置化框架

## 重难点1：通过领域模型架构设计提效保质，沉淀80%共性，支持20%特性拓展

<div class='original-text'>
这个框架最核心的就是80/20原则。我们当时发现一个问题，团队做管理后台项目时，大部分工作都是重复的。比如做个用户管理页面，要有列表、搜索、新增、编辑、删除，下一个做订单管理，又是同样的这一套。每个项目都要重复做，既浪费时间又容易出错。所以我就想，能不能把这些重复的工作沉淀下来？我设计了一套基于JSON Schema的配置协议。简单说就是，开发者只需要告诉框架"我这个页面有哪些字段，每个字段是什么类型，在哪里展示"，框架就自动生成完整功能。比如说，以前做一个用户管理页面要写2000多行代码，现在只要写一份配置，描述清楚用户名是文本、角色是下拉框、年龄是数字，该在表格显示还是在表单显示，配置完了，整个页面就出来了。列表、搜索、新增编辑、数据校验，全是自动生成的。这样一个标准页面从2天变成2小时。但是呢，业务总有特殊需求，不可能所有东西都靠配置解决。所以我设计了三层扩展机制：简单的需求在配置里加点东西就行，复杂一点的可以写自定义组件插进去，再复杂的就完全自己写，框架不限制你。这就是20%的扩展能力。实际效果很明显，我们团队开发效率提升了5到10倍，代码量少了70%，bug率降了一半，新人上手时间从一周缩短到一天。</div>
<div class="memory-aid">
  <div class="core-logic">💡 核心逻辑：80%重复CRUD → JSON-Schema配置驱动自动生成 → 20%特殊需求通过三层扩展灵活应对</div>
  <ul>
    <li><strong>背景：</strong>后台管理系统大量重复开发（列表、搜索、表单、校验）</li>
    <li><strong>方案：</strong>JSON Schema配置协议，描述字段类型与展示，前后端共用</li>
    <li><strong>80%：</strong>标准化增删改查 → 配置自动生成</li>
    <li><strong>20%：</strong>特有业务逻辑 → 三层扩展（轻量函数、插入组件、完全自定义）</li>
    <li><strong>一致性：</strong>后端定义核心Schema，前端自动补充UI并渲染，校验规则统一</li>
    <li><strong>效果数据：</strong>效率提升5~10倍，代码量减70%，Bug率降50%，新人上手1天</li>
  </ul>

### 问题1：你说的80%和20%具体指什么？ {#q1-1}

### 问题2：你这个配置方式和传统开发比，好在哪？ {#q1-2}

<div class="original-text">
主要三个优势。

第一个是<strong>统一</strong>。传统开发有个特别头疼的问题，前端写了一遍校验规则，后端又写一遍，两边经常对不上。比如前端限制年龄18到120岁，后端写的是0到100岁，这一联调就出问题了。我这个方案是<strong>前后端用同一份配置</strong>，定义一次就够了。后端说年龄是18到100，前端的表单自动就是这个规则，不会不一致。

第二个是<strong>快</strong>。以前做一个用户管理页面，写页面结构、样式、表单校验、表格分页、搜索功能、后端接口、参数校验，再加上联调修bug，怎么也得两天。现在写一份配置，一两个小时就搞定了。效率提升了<strong>10倍不夸张</strong>。

第三个是<strong>稳定</strong>。框架生成的代码都是经过充分测试的，不会出低级错误。而且自动处理各种边界情况，比如空值、特殊字符这些。我们统计过，bug率比以前降了<strong>一半</strong>。而且代码风格统一，后期维护也方便，不会出现每个人写的不一样的情况。同时还没有牺牲灵活性，通过各种hook能力，依旧保持可拓展性。

</div>

<div class="memory-aid">
  <ul>
    <li><strong>统一：</strong>前后端同一份Schema，避免校验规则不一致</li>
    <li><strong>快：</strong>配置代替手写代码，页面开发从**2天→2小时**</li>
    <li><strong>稳定：</strong>框架生成代码经过测试，边界处理完善，Bug率减半</li>
    <li><strong>灵活：</strong>通过hook/扩展保持定制能力，未牺牲灵活性</li>
  </ul>
</div>

### 问题3：如果业务需求很特殊，你的配置搞不定怎么办？ {#q1-3}

<div class="original-text">
这是肯定的，不可能所有场景都用配置解决。所以我设计了<strong>渐进式的扩展机制</strong>，根据复杂度选择不同方式。

第一种，<strong>小调整</strong>。比如某个字段要特殊展示，状态字段要显示成功失败带个图标，这种在配置里加个自定义渲染函数就行了，很轻量。

第二种，<strong>中等复杂度</strong>。比如需要批量导入功能，表格自带的功能不够用。那就自己写个批量导入的组件，然后在配置里指定一下，点某个按钮时弹出这个组件。这样既用了框架的能力，又插入了自己的逻辑。

第三种，<strong>完全自定义</strong>。比如要做个复杂的订单流程页面，有各种状态流转、可视化展示，框架的配置模式明显不适合。那就用框架的自定义模式，整个页面完全自己写，框架不限制你。

我的理念是，<strong>框架提供能力，不做限制</strong>。简单的场景用配置，效率最高；复杂的场景可以扩展，甚至完全绕过框架。这样80%场景用配置提效，20%场景能灵活应对，两边都不耽误。

实际使用下来，<strong>80%的需求用配置就够了，15%的需求用组件扩展，只有5%真正需要完全自定义</strong>。

</div>

<div class="memory-aid">
  <ul>
    <li><strong>三层渐进扩展：</strong>① 轻量：自定义渲染函数；② 中等：插入自定义组件；③ 重度：完全自定义页面</li>
    <li><strong>原则：</strong>框架提供能力，不做限制，可完全绕过</li>
    <li><strong>实际分布：</strong>80%配置解决，15%组件扩展，5%完全自定义</li>
  </ul>
</div>

### 问题4：落地过程中遇到的最大困难是什么？怎么解决的？ {#q1-4}

<div class="original-text">
最大的困难是<strong>前后端配置的平衡问题</strong>。一开始我想的很简单，前后端用同一份配置嘛，就写在一个文件里。结果发现不行，因为后端只需要知道数据类型和校验规则就够了，但前端还得知道用什么组件展示、提示文字写什么、样式怎么配。如果全塞在一份配置里，后端那边一大堆用不着的字段，特别冗余。如果分开维护，又失去了"单一数据源"的优势，还是会出现不一致的问题。

后来我的解决方案是<strong>分层</strong>。后端维护核心配置，就是数据类型、校验规则这些，这是权威数据源。前端通过接口拿到核心配置后，<strong>自动补充UI配置</strong>。比如看到这个字段有枚举值，就自动用下拉框，没有枚举值就用输入框。大部分情况自动推断就够用了，特殊情况可以手动覆盖。

这样做的好处是：后端维护最核心的规则，不冗余；前端自动生成UI配置，省事；需要定制时还能手动调整；保证了一致性，又避免了冗余。这个方案落地后效果很好，既解决了一致性问题，又保持了灵活性。

</div>

<div class="memory-aid">
  <ul>
    <li><strong>难题：</strong>前后端配置冗余与不一致</li>
    <li><strong>解法：</strong>分层设计 —— 后端维护核心Schema（数据/校验），前端自动推断UI（枚举→下拉框），特殊情况手动覆盖</li>
    <li><strong>效果：</strong>单一数据源，一致性+灵活性兼顾，消除冗余</li>
  </ul>
</div>

### 问题5：如果让你优化这个架构，会从哪入手？ {#q1-5}

<div class="original-text">
基于过去使用经验，我会从三个方向优化。

第一个是加<strong>TypeScript支持</strong>。现在配置是纯JavaScript对象，写的时候没有提示，容易拼错字段名或者写错类型。如果加上TypeScript，开发时就有完整的代码提示，拼错了也会马上报错，开发体验和质量都会提升。

第二个是支持<strong>配置联动</strong>。现在的配置是静态的，比如我想实现"当角色选管理员时，才显示权限字段"，这种动态的逻辑做不了。如果能支持配置里写函数，根据其他字段的值动态控制显隐、校验规则，能覆盖更多场景。

第三个是做<strong>可视化配置工具</strong>。现在要手写JSON配置，对非技术人员不友好。如果能做个可视化工具，左边是组件库，拖拽到画布上，右边配置属性，实时预览效果，配完了一键生成配置。这样产品经理都能参与，进一步降低门槛。

这三个方向都是基于实际使用中的痛点，如果有机会我很想把它们落地，让框架更强大更易用。当然也要看团队的资源和优先级，不一定都能做，但这是我觉得最值得投入的方向。

</div>

<div class="memory-aid">
  <ul>
    <li><strong>TS支持：</strong>类型提示与校验，减少拼写错误</li>
    <li><strong>配置联动：</strong>支持函数动态控制显隐/校验，覆盖联动场景</li>
    <li><strong>可视化工具：</strong>拖拽生成Schema，降低非技术人员门槛</li>
  </ul>
</div>
